# Proof Instances

Four products that already ran the Product Domain Engine, before it was named. Their existence is the empirical basis for the engine — the engine is not a hypothesis. It was identified by observing the recurring pattern across these four cases.

Each case below records: domain, formalization density, rhetorical-mode scores, what it proved, what it taught the engine.

## Instance 1 — `public-record-data-scrapper`

**Status:** GRADUATED  
**Organ traversal:** I → III → IV → V

### Domain
UCC public records across 50 U.S. states.

### Formalization density
- 60+ collection agents (one per jurisdiction)
- 2,055 tests
- Live deployment (Netlify)
- Paying subscribers
- Flagship README: 4,455 words

### Rhetorical-mode scores
| Mode | Score | Evidence |
|------|:-----:|----------|
| Logos | ■■■ | 60 agents, data schemas, normalization pipelines |
| Ethos | ■■■ | Flagship README, GRADUATED status, paying subscribers, deployment URL |
| Pathos | ■ | Functional B2B tool — minimal narrative; pathos was not the lever |
| Kairos | ■ | Steady B2B demand; no wave to ride; kairos is "always" |

**Dominant mode:** logos + ethos (density)

### What it proved
**Density creates gravity.** The 60 collection agents and 2,055 tests created an audience-attracting mass without any pathos investment. People found the project because it existed at sufficient density. The formalization *was* the marketing.

### What it taught the engine
1. Some domains do not need pathos. B2B utility lives on ethos density alone.
2. The flagship README is the highest-leverage ethos artifact — it converts logos mass into audience-visible credibility.
3. Test count is a credibility signal. "2,055 tests" tells the audience nothing technical; it tells them the system is serious.

### Engine-relevant pattern
**The Density Principle** — applicable to all domains: build enough real structure and traffic comes. Recorded in `references/cross-fertilization.md` (cross-domain pattern registry).

## Instance 2 — `styx-behavioral-economics-theory` → `styx-behavioral-art` → `peer-audited--behavioral-blockchain`

**Status:** Active, organ-traversing  
**Organ traversal:** I → II → III (full chain)

### Domain
Behavioral accountability via loss aversion and peer audit.

### Formalization density
- 9 theorem proofs
- Loss aversion coefficient: 1.955 (formal)
- 499+ tests
- 8 validation gates
- 4-platform implementation: NestJS + Next.js + React Native + Tauri

### Rhetorical-mode scores
| Mode | Score | Evidence |
|------|:-----:|----------|
| Logos | ■■■ | Dissertation-grade theory, theorem proofs, 4-platform deployment |
| Ethos | ■■ | Formal thesis, peer-audited design, deployment in progress |
| Pathos | ■ | Behavioral-art repo exists but emotional narrative is not customer-facing yet |
| Kairos | ■■ | Behavioral econ + DeFi both rising; peer accountability trending (BeReal, Strava) |

**Dominant mode:** logos (theory)

### What it proved
**Theory → Art → Commerce is a real pipeline.** Styx is the first product to traverse the full ORGANVM organ chain (I → II → III) end-to-end. Formal proofs derived art derived products. The traversal pattern is operational, not hypothetical.

### What it taught the engine
1. Some domains require deep theory before any product surface. Skipping Tier 2 would have produced a hollow shell.
2. The organ chain is real. Theory (I) → Art (II) → Commerce (III) is a generative sequence, not three independent products.
3. Theorem proofs are ethos artifacts. The audience does not read the proofs, but the *fact that proofs exist* lifts the trust ceiling.

### Engine-relevant patterns
**Theorem-Proving Methodology** — applicable to any domain that makes truth-claims (e.g. hokage-chess's title-scoring formula could be validated theorem-style). Recorded in cross-fertilization registry.

**Multi-Organ Traversal Scoring** — the engine evaluates which organs each domain needs. Some domains (public-record-data-scrapper) skip organs; some (styx) traverse the full chain.

## Instance 3 — `sovereign-systems--elevate-align` (Maddie)

**Status:** LOCAL, client engagement active  
**Organ traversal:** I (light) → II → III

### Domain
Health/wellness practitioner with multi-domain brand. Hub-and-spoke topology: `elevatealign.com` (hub) + `stopdrinkingacid.com` (water/physical sovereignty) + `eaucohub.com` (financial sovereignty business arm).

### Formalization density
- 13-node spiral model (Three.js)
- 4-pillar system (Physical, Inner, Identity, Financial)
- Hub-and-spoke topology across 3 domains
- 40 atomic brand components extracted
- Astro 5 + Tailwind 4 + Cloudflare Pages

### Rhetorical-mode scores
| Mode | Score | Evidence |
|------|:-----:|----------|
| Logos | ■■ | Architecture exists, spiral math works, hub.config.ts is single source of truth |
| Ethos | ■ | LOCAL status, not yet at custom domain, client-feedback gated |
| Pathos | ■■ | Spiral is emotionally resonant, brand is warm — landing page conversion-tuning pending |
| Kairos | ■ | Wellness practitioners over-served on websites; kairos depends on Maddie's readiness |

**Dominant mode:** balanced (slight pathos lead)

### What it proved
**One practitioner's knowledge maps to multiple audience entry points through topology.** A hub-and-spoke architecture lets a single practitioner address multiple verticals without splintering brand. The 13-node spiral encodes the practitioner's full taxonomy as a navigable structure.

### What it taught the engine
1. Client-driven domains may need pathos before architecture. Maddie wanted to *see* the spiral before architecture was locked. The engine adapted (Tier 4 first, Tier 3 follows).
2. Topology is a formalization output. The hub-and-spoke pattern is a logos artifact (it has rules) that functions as pathos externally (it makes the brand *navigable* and emotionally legible).
3. Physical-product derivation chains apply: Maddie's protocols can become physical journals / cards / cycle trackers, derived from the formalization.

### Engine-relevant patterns
**Hub-and-Spoke Topology** — applicable to any domain with multiple audience entry points. hokage-chess's future multi-domain (`hokagechess.com` / `thedojo.gg` / `hokagecourses.com`) inherits this pattern.

**Brand Embedding Structure** — voice + naming + visual + narrative as a four-layer brand stack, recorded in the Phase 4 specification.

## Instance 4 — `hokage-chess` (Rob)

**Status:** LOCAL, client engagement active  
**Organ traversal:** I (light) → II → III → VI → VII

### Domain
Chess improvement at the amateur level, documented as entertainment. The hero's journey applied to the rating ladder.

### Formalization density
- TypeScript engine: title scorer, narrative framework, analytics, growth model
- Next.js 16 + Tailwind v3 landing page (9 sections)
- 7 market gaps identified
- 7 unique courses (Naruto-themed price ladder $0–$500)
- Trademark research complete
- Full business model documented

### Rhetorical-mode scores
| Mode | Score | Evidence |
|------|:-----:|----------|
| Logos | ■■ | Engine exists with types and tests; not yet deployed |
| Ethos | ■ | 99 subs / 322 videos — credibility building, not yet at proof-point |
| Pathos | ■■■ | Strongest pathos of any instance — underdog narrative, shōnen arc, Naruto identity, emotional landing page |
| Kairos | ■■ | Chess content market post-pandemic normalized; "improvement journey" niche rising; kairos OPEN for community |

**Dominant mode:** pathos (narrative)

### What it proved
**The full Product Domain Engine can be run in one session.** Market-gap analysis → business model → landing page → course catalog → trademark research in 24 hours. The engine is fast when the domain is well-known to the operator.

### What it taught the engine
1. Pathos-dominant domains lead with narrative. The Naruto identity (Genin → Hokage rank progression) IS the formalization, not decoration.
2. The TypeScript engine modules (`title-scorer.ts`, `analytics.ts`, `growth.ts`) are simultaneously logos (typed business logic) and the autopoietic observation instruments — they will measure the deployed surface and feed back into formalization.
3. Single-session formalization is feasible for domains with strong pre-existing operator knowledge (Rob's 322 videos; Anthony's chess familiarity).

### Engine-relevant patterns
**Ki-Shō-Ten-Ketsu Narrative Framework** — four-act narrative structure (setup → development → twist → resolution) — applicable to how styx presents its thesis, how any domain structures a pitch deck.

**Naruto Rank System** — taxonomic / community-tier pattern (Genin → Chunin → Jonin → Hokage) — applicable to any domain with skill bands (Discord roles, course tiers, achievement systems).

**$0–$500 Price Ladder** — 8-tier pricing covering freemium → community → digital products → cohort → premium service. Mode-tagged in `composition-matrix.md`.

### Key observation

**Each instance leads with a different rhetorical mode.** The engine does not prescribe which mode dominates — the *domain's nature* determines it. B2B utility leads with logos. Theoretical product leads with logos. Client-brand leads balanced/pathos. Entertainment-community leads with pathos.

The engine's job is not to pick the lead mode. The engine's job is to ensure **all four modes are addressed** *regardless* of which one leads.

## Verification Of The Engine

Run `scripts/domain-audit.sh` against all four instances. The dominant-mode result should match the table above (with the documented caveat that the script reads structure, not voice — pathos detection is biased toward `brand/` directories and narrative files, so hokage-chess will under-score on pathos in the script even though it is pathos-dominant in lived practice).

The audit is a *coverage* check. Quality assessment requires human reading.
