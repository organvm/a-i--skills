# Triple-Reference Protocol

## Overview

The Triple-Reference axiom requires that every governed repository maintains convergence across three locations:

1. **IRF (Identity Reference File)** — On-disk identity declaration
2. **Repository Remote** — Git remote origin URL
3. **GitHub Tracking Issue** — Issue tracking the governance state

Identity is established by convergence. If any reference is missing or inconsistent, the repository is in violation.

## IRF (Identity Reference File)

### Location

`IRF.md` in the repository root.

### Required Fields

```markdown
# Identity Reference File

- **Repository:** <owner>/<repo>
- **Governance State:** <LOCAL|CANDIDATE|PUBLIC_PROCESS|GRADUATED>
- **Governance Version:** <integer>
- **GitHub Issue:** <full URL to tracking issue>
- **Remote Origin:** <git remote URL>
- **Triple-Reference Checksum:** <sha256 of concatenated IRF + seed.yaml + remote URL>
- **Last Verified:** <ISO 8601 timestamp>
```

### Checksum Calculation

The triple-reference checksum is computed as:

```
sha256sum(IRF.md content + seed.yaml content + git remote get-url origin)
```

This ensures that any change to identity, state, or remote requires a checksum update.

## Repository Remote

### Validation

- Must be a valid git remote URL (HTTPS or SSH)
- Must resolve to an existing repository on the remote host
- Must match the `Repository` field in `IRF.md`

### Commands

```bash
git remote get-url origin
git ls-remote --heads origin  # Verify remote is accessible
```

## GitHub Tracking Issue

### Purpose

The tracking issue documents the governance migration history, audit results, and transition approvals for the repository.

### Required Content

- Title: "Governance Migration: <repo> → <target state>"
- Body must include:
  - Source state and target state
  - Migration manifest reference
  - Approval timestamp
  - Audit results (pass/fail)
  - Link to `IRF.md`

### Creation

Use `gh issue create` or manual creation via GitHub UI. The `MISSING_REFERENCES_CMDS.sh` and `MISSING_REFERENCES.md` files generated by the executor provide the exact commands/steps.

## Consistency Checks

The `verify_triple_reference.py` auditor validates:

1. `seed.yaml` exists and contains a valid `state` enum value.
2. `IRF.md` exists and contains all required fields.
3. `git remote origin` is configured and accessible.
4. GitHub tracking issue URL in `IRF.md` resolves to an existing issue.
5. The `state` field in `seed.yaml` matches the `Governance State` in `IRF.md`.
6. The `Remote Origin` in `IRF.md` matches `git remote get-url origin`.
7. The triple-reference checksum is valid.

## Signal Protocol

When a repository achieves triple-reference alignment:

1. The auditor exits with code 0.
2. The rollback manifest marks the operation as verified.
3. The GitHub tracking issue is updated with a "verified" label.

When a repository fails triple-reference validation:

1. The auditor exits with code 1.
2. A `violation_report.md` is generated with actionable findings.
3. The GitHub tracking issue is updated with a "violation" label.
4. If the repository was GRADUATED, it is degraded to PUBLIC_PROCESS.
