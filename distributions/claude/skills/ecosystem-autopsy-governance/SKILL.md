# Ecosystem Autopsy & Governance Migration

Enforce ecosystem-wide governance via the 4-tier model (LOCAL, CANDIDATE, PUBLIC_PROCESS, GRADUATED) and the Triple-Reference axiom. Map workspaces, evaluate governance states, and safely coordinate migration of repositories with defensive systems engineering, batch atomicity, idempotent rollbacks, and explicit collision detection.

## When to Use

- Audit the ecosystem for governance compliance
- Plan and execute repository migrations between governance states
- Verify triple-reference alignment across repositories
- Generate migration manifests with dependency-aware priority ordering

## Prerequisites

- Python 3.11+ with `pyyaml` installed (`pip install pyyaml`)
- `git` CLI available
- `gh` CLI (optional, for automated GitHub issue creation)

## Phase Gates

Execute phases sequentially. Do not skip phases. Do not proceed to Execute without explicit user approval.

### Phase 1: Map

Run the autopsy scanner to map the ecosystem:

```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
python3 "$SCRIPT_DIR/scripts/ecosystem_autopsy.py" \
  --workspace ~/Workspace ~/Code ~/Documents \
  --output autopsy-report.json \
  --manifest migration_manifest.json
```

Review the output:
- `autopsy-report.json` — ecosystem stats, violations, orphans
- `migration_manifest.json` — deterministic migration plan

### Phase 2: Plan

Present the migration manifest to the user:

1. Summarize the number of operations by type (`git_mv`, `mv`, `init`, `noop`)
2. Highlight any violations or unresolved dependencies
3. Show the priority ordering and topological sort rationale
4. Present the dry-run output:

```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
python3 "$SCRIPT_DIR/scripts/migration_executor.py" \
  --manifest migration_manifest.json
```

### Phase 3: Halt — Wait for User Approval

<wait_for_user>

Do not proceed until the user explicitly approves the migration plan. The user may:
- Approve as-is
- Request modifications to the manifest
- Reject and abort

### Phase 4: Execute

Upon explicit user approval, run the executor in `--execute` mode:

```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
python3 "$SCRIPT_DIR/scripts/migration_executor.py" \
  --manifest migration_manifest.json \
  --execute
```

The executor will:
- Write `rollback_manifest.json` before any mutation
- Execute operations sequentially with batch atomicity
- Auto-rollback on failure
- Generate `MANUAL_INTERVENTION_REQUIRED.md` for depth changes >= 1
- Generate `MISSING_REFERENCES.md` and `MISSING_REFERENCES_CMDS.sh` for repos lacking triple-reference

### Phase 5: Audit

Run the triple-reference auditor on migrated repos:

```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
python3 "$SCRIPT_DIR/scripts/verify_triple_reference.py"
```

For ecosystem-wide verification:

```bash
python3 "$SCRIPT_DIR/scripts/verify_triple_reference.py" --ecosystem-wide
```

### Phase 6: Human Commit Presentation

Run `git status` for each migrated repository and present the staged changes to the user. Invite the user to review and commit.

**Git Policy:** Do NOT run `git commit` or `git push` automatically. Only `git add` and `git status` are permitted. The human commits. Always.

## Rollback

If the user needs to reverse a migration:

```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
python3 "$SCRIPT_DIR/scripts/migration_executor.py" \
  --manifest migration_manifest.json \
  --undo
```

The `--undo` flag is idempotent — running it twice does nothing on the second run.

## Data Contracts

### seed.yaml Schema

```yaml
state: LOCAL                          # Required. One of: LOCAL, CANDIDATE, PUBLIC_PROCESS, GRADUATED
governance_version: 1                 # Required. Integer.
dependencies:                         # Optional. Relative paths to dependency repos.
  - ../base-repo
description: ""                       # Optional.
owner: ""                             # Optional.
```

### migration_manifest.json Schema

```json
{
  "ecosystem_root": "/path/to/workspace",
  "generated_at": "2026-05-17T...",
  "operations": [
    {
      "operation_id": "op_001",
      "priority": 100,
      "source_path": "./legacy_app",
      "target_path": "./CANDIDATE/legacy_app",
      "type": "git_mv",
      "depth_change": 1,
      "is_untracked_dir": false,
      "requires_init": false
    }
  ]
}
```

**Type enum:** `git_mv` | `mv` | `init` | `noop`

### rollback_manifest.json Schema

```json
{
  "batch_id": "batch_1698421000",
  "status": "COMPLETED",
  "operations": [
    {
      "rollback_id": "rb_001",
      "forward_operation_id": "op_001",
      "inverse_action": "git mv ./CANDIDATE/legacy_app ./legacy_app",
      "pre_state_hash": "sha256:...",
      "timestamp": "2026-05-17T...",
      "rolled_back": false
    }
  ]
}
```
