     Approach (sketch вҖ” to be refined after exploration)

     Three layers, each minimal-generative (Universal Rule #36 вҖ” seed, not
     specification):

     Layer 1: Audience tagging on internal artifacts (substrate)

     Every internal artifact gets frontmatter declaring:
     - audiences: вҖ” [internal, client:rob, client:maddie] (multi-target allowed)
     - strata: вҖ” which 8-strata layer this artifact lives in
     - register: вҖ” technical | strategic | tactical | narrative
     - client_render: вҖ” eli5 | tldr | full | skip (the "if logical" gate)
     - bridge_to: вҖ” links to client-language analogues that already exist

     This makes audience a property of the artifact, not a downstream decision.

     Layer 2: Persona-bound translation engine

     Each persona (~/Documents/personas/{name}.md) carries:
     - lexicon: вҖ” preferred terms (Rob: "drill", "rep", "boss battle"; Maddie:
     "embodied", "ritual", "lineage")
     - analogies: вҖ” domain-native metaphors for technical concepts (funnel вҶ’
     chess opening; substrate вҶ’ training plan)
     - forbidden_terms: вҖ” register-violating jargon to suppress
     - cognitive_load: вҖ” depth tolerance per surface (storefront top = TLDR;
     deep page = ELI5; appendix = full)
     - voice_anchor: вҖ” which corpus to draw rhythm from (Rob's YouTube
     transcripts; Maddie's intake doc)

     A render pipeline takes (artifact, persona) вҶ’ translated surface. Built as
     a skill (personalized-storefront-render) that calls voice-enforcement +
     persona profile + artifact frontmatter.

     Layer 3: Storefront generator (deploy surface)

     Per (client Г— project) storefront, structured as:
     - / вҖ” TLDR home (1-screen what-we're-doing-together)
     - /work вҖ” gallery of client-tagged artifacts, ELI5 view by default, "show
     technical view" toggle
     - /decisions вҖ” decision log in their language (why we chose this)
     - /next вҖ” what's pending from them (consolidates ask packets like MD-1..7)
     - /glossary вҖ” domain-translation table

     Static site (Astro or Next.js) consuming the audience-tagged corpus +
     persona profile, deployed per-client subdomain (e.g., rob.4jp.io,
     maddie.4jp.io) gated by simple shared link.

     Lifecycle (Universal Rule #6 вҖ” everything is a loop)

     Internal artifact created/updated
       вҶ’ audience frontmatter check (skip if untagged or client_render: skip)
       вҶ’ translation engine renders client view per matched persona
       вҶ’ storefront rebuild + deploy
       вҶ’ user notified of diff to share/announce

     Plus a backfeed loop: client questions/feedback on storefront вҶ’ atoms in
     IRF.
